## Critical Architecture Rules

### Nhost/Hasura-First Architecture
- ALL database operations go through Hasura GraphQL API
- Authorization is configured in Hasura permissions, NOT in application code
- Use `itap` schema for all ITAP tables (separated from Nhost's `auth` and `storage` schemas)
- Nhost Auth provides JWT with `x-hasura-user-id`, `x-hasura-default-role`, `x-hasura-allowed-roles`

### Schema Conventions
- All ITAP tables live in the `itap` schema
- Table names: `itap_{entity}` (e.g., `itap_identities`, `itap_profiles`)
- Column names: `snake_case`
- Primary keys: `id` as `uuid` with `gen_random_uuid()` default
- Timestamps: `created_at` and `updated_at` as `timestamptz` with defaults
- Foreign keys: `{related_entity}_id` (e.g., `profile_id`)
- Soft delete: `deleted_at` column where needed (nullable `timestamptz`)

### Permission Model (Nhost Auth Roles)
- `admin` -- full access to all tables
- `user` -- access to own data and shared resources
- Session variables from Nhost Auth:
  - `x-hasura-user-id` -- authenticated user's ID
  - `x-hasura-default-role` -- user's default role
  - `x-hasura-allowed-roles` -- list of allowed roles

## Migration Workflow (MANDATORY)

### Creating a New Table
```bash
# 1. Create migration directory with timestamp
TIMESTAMP=$(date +%s%3N)
mkdir -p apps/nhost/nhost/migrations/default/${TIMESTAMP}_create_table_itap_{entity}

# 2. Write up.sql (create)
cat > apps/nhost/nhost/migrations/default/${TIMESTAMP}_create_table_itap_{entity}/up.sql << 'SQL'
CREATE TABLE itap.{entity} (
    id uuid NOT NULL DEFAULT gen_random_uuid(),
    -- columns here
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY (id)
);
COMMENT ON TABLE itap.{entity} IS 'Description of the entity';
SQL

# 3. Write down.sql (rollback)
cat > apps/nhost/nhost/migrations/default/${TIMESTAMP}_create_table_itap_{entity}/down.sql << 'SQL'
DROP TABLE IF EXISTS itap.{entity};
SQL

# 4. Apply locally
make hasura_apply

# 5. Update metadata (track table, add permissions)
# Edit apps/nhost/nhost/metadata/databases/default/tables/itap_{entity}.yaml
```

### Adding a Column
```bash
TIMESTAMP=$(date +%s%3N)
mkdir -p apps/nhost/nhost/migrations/default/${TIMESTAMP}_alter_table_itap_{entity}_add_column_{column}

# up.sql
ALTER TABLE itap.{entity} ADD COLUMN {column_name} {type} {constraints};

# down.sql
ALTER TABLE itap.{entity} DROP COLUMN IF EXISTS {column_name};
```

### Adding a Foreign Key
```bash
TIMESTAMP=$(date +%s%3N)
mkdir -p apps/nhost/nhost/migrations/default/${TIMESTAMP}_set_fk_itap_{entity}_{column}

# up.sql
ALTER TABLE itap.{entity}
  ADD CONSTRAINT {entity}_{column}_fkey
  FOREIGN KEY ({column})
  REFERENCES itap.{related_entity}(id)
  ON UPDATE CASCADE ON DELETE SET NULL;

# down.sql
ALTER TABLE itap.{entity} DROP CONSTRAINT IF EXISTS {entity}_{column}_fkey;
```

## Permission Configuration

### Metadata File Format
```yaml
# apps/nhost/nhost/metadata/databases/default/tables/itap_{entity}.yaml
table:
  schema: itap
  name: {entity}
object_relationships:
  - name: profile
    using:
      foreign_key_constraint_on: profile_id
array_relationships:
  - name: identities
    using:
      foreign_key_constraint_on:
        column: profile_id
        table:
          schema: itap
          name: identities
select_permissions:
  - role: user
    permission:
      columns:
        - id
        - name
        - status
        - created_at
        - updated_at
      filter: {}
      allow_aggregations: true
  - role: admin
    permission:
      columns: "*"
      filter: {}
      allow_aggregations: true
insert_permissions:
  - role: admin
    permission:
      columns:
        - name
        - status
      check: {}
update_permissions:
  - role: admin
    permission:
      columns:
        - name
        - status
      filter: {}
delete_permissions:
  - role: admin
    permission:
      filter: {}
```

### Permission Rules for Common Patterns

**Public read, admin write:**
```yaml
select_permissions:
  - role: user
    permission:
      columns: [id, name, status, created_at]
      filter: {}
insert_permissions:
  - role: admin
    permission:
      columns: [name, status]
      check: {}
```

**Owner-only access:**
```yaml
select_permissions:
  - role: user
    permission:
      columns: [id, name, created_at]
      filter:
        user_id:
          _eq: X-Hasura-User-Id
```

## Quality Checklist

Before completing any migration:
- [ ] Migration files have both `up.sql` and `down.sql`
- [ ] `down.sql` correctly reverses `up.sql`
- [ ] Primary key is `uuid` with `gen_random_uuid()`
- [ ] `created_at` and `updated_at` timestamps included
- [ ] Foreign keys have proper ON DELETE/UPDATE actions
- [ ] Indexes on frequently queried columns and foreign keys
- [ ] Table in `itap` schema (not `public`)
- [ ] Permissions configured for `user` and `admin` roles
- [ ] Relationships defined (object and array)
- [ ] Migration applies cleanly: `make hasura_apply`
- [ ] No breaking changes to existing queries

## DEMO Project Guidelines

- Keep schema simple -- avoid premature normalization
- Use `jsonb` columns for flexible/evolving data (e.g., `attributes`, `metadata`)
- Skip audit tables and complex triggers for now
- Use Hasura's built-in `created_at`/`updated_at` tracking
- Focus on the core entities needed for the demo
- Defer complex row-level security until after MVP
